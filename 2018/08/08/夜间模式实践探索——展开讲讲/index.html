夜间模式实践探索——展开讲讲

Aug 8, 2018

承接上文，调用 setDefaultNightMode 来切换日间和模式，对于已经启动好的页面，调用 setLocalNightMode并对布局内的 View 重新渲染颜色值。文本讲展开讲讲，具体是怎么实现的

整理颜色值

工欲善其事，必先利其器。实践夜间模式的第一步，是整理应用内的颜色值。比如，除非特殊的业务要求，不然页面的背景色应该都是一个色值，默认的字体颜色也应该是同一种黑。然后把整理出来的颜色值和夜间模式的颜色值对应起来

你可以把基本的颜色值放到 base_colors 中，而需要夜间模式的颜色值使用 night_ 开头命名

新建一个文件夹命名为 values-night，把夜间模式中的颜色值放在这个文件中

注意！它们的颜色值命名必须相同！当你切换夜间模式后，系统就会使用 values-night 中的颜色值来渲染新的页面~

这样，除了当前「活着的」的页面，其他新创建的页面都会使用夜间模式的颜色值来创建。所以剩下的 5% 就是要解决「活着的」的页面。如果你仔细观察有夜间模式的 App，你会发现他们无一例外都把页面模式的开关放在一级页面中，这样就就减少了「活着的」页面的处理。

因为在实际业务中，「活着的」Activity 只有 MainActivity。首先，我们要在 AndroidMainfest 中给他设置 android:configChanges="uiMode"，然后在切换的时候使用 setLocalNightMode 的方法。这个方法一般来说会配合 activity # recreate 使其生效。这样对于用户来说体验较差，而且市面上也很少应用会使用这样的方式，所以要采用重新刷新 UI 的方式

NightView

为了刷新 UI，定义了接口 NightView

接着，创建 NightTextView、NightFragment、NightRelativeLayout…等 MainActivity 中有用到的 View，让它们实现 NightView。每个 NightView 由 NightViewAttacher 来处理内部逻辑

根据 View 的特点在 resetNightColors 中重渲染色值即可。setNightCallback 是一种特殊的情况，因为涉及到 NightAttacher 的逻辑，所以会在 NightAttacher 中介绍

看到这里，你可能眉头一皱：那是不是要把所有 MainActivity 的 View 全部替换成 NightView？其实如果你们项目已经封装了自定义 View，那只需要让它们实现 NightView 接口，并创建 NightViewAttacher，剩下的工作桥接给它就行了。如果你们还在用原生的 View，那么可以使用 LayoutInflaterCompat # setFactory2 对 xml 中的 View 进行替换。

不过，代码里的创建的 View 你还是得手动替换成 NightView，比如

val textView = NightTextView(context)

NightViewAttacher

在 attrs 文件中声明原生属性

这样，我们就可以在 NightViewAttacher 中通过 getResourceId 获得原生颜色属性，这个属性是 @colorRes，通过 ContextCompat 生成相应的色值。在夜间模式下，系统会去使用 night 文件中定义的色值

注意，如果是通过 TypeArray # getColor 拿到的色值不会根据夜间模式的状态去切换不同色值，只有通过 ContextCompat # getColor 生成的色值才会生效

在 NightAttacher 中，我们只能获得 xml 中定义的色值。如果是代码中设置的颜色，就会导致切换的时候无法改变。基于这个原因，我们定义了 setNightCallback 的方法，把设置颜色的方法放到 callback 内部做。在 NightView # resetNightColors 中把 callback 中的方法再执行一遍，达到替换颜色的目的

我们可以看到，内部的实现无非就是把 callback 缓存起来，然后在 toggleCallback 中执行。toogleCallback 会在 NightView # resetNightColors 中执行。值得注意的是， setNightCallback 的 callback 会在设置进去的时候调用一次，便于替换外部的方法

加入夜间模式后，在代码中设置颜色的方法被挪到 setNightCallback 的回调中执行，因为 setNightCallback 的回调默认会调用一次，所以对原先逻辑的改动就变得更少了

resetView

对主页面进行递归遍历，实现主页面所有元素重新渲染色值，从而实现夜间模式的切换

如果 view 实现了 NightView，那么执行 resetNightColors 重新渲染色值；如果 view 是 ViewGroup ，那么对其它的子 View 进行遍历执行 resetView，递归下去就可以把 view 的色值都重置

因为不使用默认的 ActionBar，所以可以通过上述 MainActivity # getRootView 来获取根 View

RecyclerView

因为 RecyclerView 布局复用的原因，resetView 会漏掉复用的布局。ItemView 不可见却缓存在 RecyclerView 的内存中，所以它们的布局在切换夜间模式的时候不会被重新渲染，这导致 RecyclerView 会在滚动的时候复用原先的没被重新渲染新色值的布局，最终导致了页面异常

解决的方法是把 RecyclerView 中缓存的复用布局清空掉，让 RecyclerView 使用重新渲染好颜色的 ItemView

创建 NightRecyclerView 继承 RecyclerView 并实现 NightView，在NightView 的 resetNightColors 中通过反射调用 RecyclerView 的 mRecycler # clear 并调用 recycledViewPool # clear。

禁用夜间模式

当我们实现完这些后，可能会出现一些页面根本不需要夜间模式，那么针对这些页面我们也要做特殊处理。对此，我们定义了 DisableNight 的接口，让不需要夜间模式的页面实现这个接口。然后判断，如果页面是 DisableNight，就在页面启动的时候调用 setLocalNightMode 设置日间模式，页面结束的时候设置回夜间模式。这看似稳得一匹，其实包含了一个 BUG —— setLocalNightMode 会影响所有「活着的」页面，导致跳转到禁用夜间模式页面时，如果上个页面刚好请求的数据返回并渲染，这些渲染的 UI 就会恢复成日间模式。

那为什么会出现这种情况，又得如何解决，我们将在下一篇「夜间模式实践探索——源码分析」中一探究竟~

总结

通过整个流程走下来，我们把最基本的夜间模式实现简要地分析了一遍。

有两个方面需要理清楚，第一，夜间模式的具体效果不难实现，通过 setDefaultNightMode 就可以轻松实现夜间模式效果。繁琐的工作在于切换夜间模式时，当前的 MainActivity 中 UI 需要重新渲染。因此，我们需要把 MainActivity 中的 View 全部替换成 NightView，然后递归遍历渲染。而对于特殊的 RecyclerView，则需要把它内部复用的布局清空掉。

第二，所有 View 拿到的 color 值都是16进制的数字，它不会因为夜间模式的改变而改变，而如果拿到它们的 colorRes，就可以让 ContextCompat # getColor 内部根据夜间模式的状态去拿不同资源的色值，从而达到改变夜间模式颜色的效果。
