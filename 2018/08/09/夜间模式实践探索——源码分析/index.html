夜间模式实践探索——源码分析

Aug 9, 2018

在上文的结尾谈到了一个新的需求，假设页面不需要夜间模式，那么调用 setLocalNightMode 的方法来改变当前的夜间模式状态来取消当前页面夜间模式，但是其实可能会导致上一个页面的 UI 出现问题。回顾一下 setLocalNightMode 的方法

我们发现，它是由当前 activity 对象中的 delegate 来调用这个方法，然而它不仅改变了自身夜间模式的状态，甚至连其它「活着的」Activity 的夜间模式状态也都改变了。这究竟是怎么一回事，带着这个疑惑，我们来分析其中的源码实现，帮助我们更好地理解这套 Android 夜间模式

首先，在 AppCompatActivity 中找到 getDelegate

代码很简单，mDelegate 通过 AppCompatDelegate # create 来获得。在 AppCompatDelegate 中，它是通过 create 来创建本身

从代码中看出，它是根据不同 Android 版本创建不同对象，但是最基本的还是 AppCompatDelegateImplV14，因为其它都是它的子类进行扩展。因为我们获得一个有用的信息，当前夜间模式的功能最低支持 API14

我们在 AppCompatDelegateImplV14 中找到了 setLocalNightMode

从代码中看出，setLocalNightMode 除了把 mode 赋值给 mLocalNightMode 外，还会调用一遍 applyDayNight 来使夜间模式生效。appDayNight 还会在 Activity 的 onCreate 和 onStart 中被调用

这里省略了一些代码，首先通过 getNightMode 和 mapNightMode 获得最终 mode

getNightMode 的代码很简单，如果 mLocalNightMode 有有效值，就使用 mLocalNightMode, 否则调用 getDefalutNightMode

getDefalutNightMode 是 AppCompatDelegateImplV14 的最初父类 AppCompatDelegate 中的方法，它返回 setDefaultNightMode 设置的 mode，即优先使用 LocalNightMode，否则就使用 DefaultNightMode

mapNightMode 只是对 mode 的类型转换，这里就不赘述了。回过头来继续看 applyDayNight，当我们把夜间模式的 mode 设置为MODE_NIGHT_YES 或者 MODE_NIGHT_NO 时，都会触发 updateForNightMode

省略了一些代码，关键的代码是 Resoures # updateConfiguration，它会更新资源文件的引用——如果 night 属性文件中含有相同命名的资源，就使用该资源，否则继续使用默认资源文件中的文件

到这里，对于之前第二个页面 setLocalNightMode 恢复日间模式，上个页面渲染的 UI 就会是日间模式的 bug 似乎就可以理解了。因为 setLocalNightMode 最终更改的是全局的资源文件的引用方式，各个页面只是用给各自的 context 修改全局的 Resoures

总结

当我们通过 setDefalutNightMode 的时候，Activity # onCreate 会调用 applyDayNight，内部会根据defaultNightMode 来确定资源引用方式；当我们切换夜间模式时，我们会调用 setLocalNightMode，这个方法内会调用 applyDayNight，内部是忽略之前的 defalutNightMode，而是用 mLocalNightMode 来确定新的资源引用方式。因为当前页面的数据已经渲染完成了，新的资源引用方式只对 setLocalNightMode 后渲染的有效，所以我们用 resetView 来重新渲染色值。
